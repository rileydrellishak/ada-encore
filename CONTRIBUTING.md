# 🌱 How to Use Ada Encore

## 🧭 What this project is

Ada Encore is a personal re-run of Ada Developers Academy Unit 1–4 curriculum, but with an experienced engineer’s perspective layered on top.

It is not:

* a new curriculum
* a portfolio project
* a tutorial series

It is:

> a structured revisit of fundamentals to rebuild clarity, confidence, and engineering habits

---

## 🎯 Core Intent

Every part of Ada Encore exists to answer one question:

> “What would Future Riley understand now that Student Riley didn’t then?”

This means:

* prioritizing clarity over novelty
* reinforcing fundamentals over expanding scope
* focusing on thinking habits, not tooling trends

---

## 🧱 Structure Overview

Each sprint contains:

* `README.md` → syllabus + mindset + navigation
* `assignment-XX.md` → core learning tasks
* `resources/` → reusable engineering tools
* `side-quests/` → optional exploration

---

## 📚 How to Work Through a Sprint

When starting a sprint:

1. Read the sprint README fully
2. Skim all assignments once (don’t dive in immediately)
3. Start with Assignment 1
4. Move sequentially unless explicitly blocked
5. Use resources when stuck, not as pre-reading
6. Do side quests only after core assignments feel stable

---

## 🧠 How to Approach Assignments

Each assignment is designed to be:

* small and focused
* conceptually isolated
* reflective, not rushed

Good workflow:

1. Restate the problem in your own words
2. Break it into smaller parts
3. Write a simple first version
4. Refactor for clarity
5. Reflect briefly
6. Move on

> Working code is not the finish line. Understandable code is.

---

## 🐛 How to Use Resources

### Debugging Workflow

Use when:

* you feel stuck
* errors feel overwhelming
* you don’t know where to start

👉 Trust the steps. Don’t skip to guessing.

---

### Code Review Checklist

Use when:

* code “works but feels messy”
* before submitting assignments
* during refactors

👉 Think: “Would another engineer understand this quickly?”

---

## ⭐ How to Use Side Quests

Side quests are optional exploration layers.

Use them when:

* you finish early
* you feel curious
* you want repetition with variation

Do NOT use them to:

* delay starting assignments
* over-optimize learning

They are:

> reinforcement, not requirement

---

## 🌿 Philosophy Reminders

### 1. Clarity > Speed

Slow thinking produces better code than fast guessing.

### 2. Simplicity is a skill

If something feels complex, it probably can be simplified.

### 3. Debugging is normal

Errors are part of programming, not a failure state.

### 4. Repetition is learning

Seeing concepts again is intentional, not redundant.

---

## 🧰 Engineering Mindset

Throughout this repo, you are practicing:

* decomposition
* abstraction
* readability
* debugging discipline
* data modeling
* incremental refactoring

These are transferable skills across all languages and frameworks.

---

## 🏁 What “done” looks like

Ada Encore is successful when:

* you feel more confident in fundamentals
* you write simpler solutions over time
* you debug with less frustration
* you explain your code more clearly
* you trust your thinking more than intuition

---

## ☕ Final Note

This project is not about proving you can code.

It is about rebuilding the feeling that:

> you understand what your code is doing, and why

That is the foundation everything else sits on.

---

## 🔁 Maintenance Philosophy

If you update Ada Encore in the future:

* prefer clarity over expansion
* remove overcomplicated additions
* keep assignments small and focused
* preserve Ada’s original scaffolding unless there is a strong reason not to
